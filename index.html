ATS System — Pura Overview
Applicant Tracking System ka complete global architecture. Har concept point-wise samjha hai.
System kya karta hai?
  • 01User (Recruiter Company) — jo hamara software khareedte hain. Inhe "Tenant" kehte hain. Jab ye pehli baar register karte hain to ye Admin ban jaate hain.
  • 02Client — Tenant ka customer. Ye company hoti hai jo jobs deti hai jaise "TCS ko 5 Java developers chahiye."
  • 03Job — Client deta hai. Ek job ke liye ek Pipeline banti hai jisme candidates track hote hain.
  • 04Pipeline — Ek hiring funnel. Stages hote hain jaise Screening → Interview → Offer → Hired. Tenant apne stages customize kar sakta hai.
  • 05Candidate — Jo log apply karte hain. Ek candidate multiple pipelines mein ho sakta hai — aur har pipeline mein uska alag stage/status hoga.
  • 06Contract — Tenant (User) aur Client ke beech agreement. Fee structure, validity sab yahan store hota hai.
"Global" ka matlab kya hai?
CONCEPT Convention over Configuration
  • →System ka default behavior itna achha ho ki 80% users ko kuch change hi na karna pade
  • →Baki 20% ke liye override karne ka clean rasta ho
  • →Ek codebase, hazaro tenants ke liye alag alag experience
PATTERN Tenant Config Object
  • →Login ke baad backend ek config object bhejta hai
  • →Pipeline stages, custom fields, feature flags — sab issi mein
  • →Frontend isi config se dynamically UI banata hai

Multi-Tenancy Architecture
Ek hi codebase, alag alag companies ke liye completely isolated data.
3 Approaches — Kaunsa best hai?
APPROACH 1
Separate Database
  • ✓Maximum isolation
  • ✗Bahut expensive
  • ✗Manage karna mushkil
APPROACH 2
Schema Per Tenant
  • ✓Strong isolation
  • ✗Schema migration complex
  • ~Medium complexity
APPROACH 3 ✓ BEST
Shared Tables + RLS
  • ✓Simple architecture
  • ✓PostgreSQL RLS = DB-level isolation
  • ✓Industry standard SaaS pattern
Hamara approach — Shared Tables + tenant_id
  • 01tenant_id column — har ek table mein hota hai. Koi bhi query chal sakti hai, sirf us tenant ka data aayega.
  • 02Row Level Security (RLS) — PostgreSQL ka built-in feature. Database level pe guarantee karta hai ki tenant A kabhi tenant B ka data nahi dekh sakta — chahe code mein galti bhi ho.
  • 03TenantGuard in NestJS — har HTTP request mein subdomain ya header se tenantId resolve hota hai aur request object pe set hota hai.
  • 04@TenantId() decorator — controller mein tenantId seedha milti hai, service ko koi extra lookup nahi karna padta.
  • 05JWT mein tenantId — login ke baad JWT token mein tenantId embed hoti hai. Har request ke saath automatically aati hai.
Tenant Resolution — Step by Step
STEP 01
Browser se request aati hai
acme.yourats.com/api/jobs — subdomain mein "acme" tenant hai
STEP 02
Next.js middleware.ts
Edge pe subdomain parse hota hai → tenantSlug header add karta hai request mein
STEP 03
NestJS TenantGuard
Header se tenantSlug leti hai → DB se tenantId fetch karti hai → req.tenantId set karti hai
STEP 04
Controller & Service
@TenantId() decorator se tenantId milti hai → har DB query mein WHERE tenant_id = ? automatically

Roles & Permissions
System roles fixed hain. Tenant apne custom roles bana sakta hai inhe copy karke.
  • 01is_system = true — Ye roles system deta hai. Tenant inhe delete ya edit nahi kar sakta.
  • 02is_system = false — Tenant ke custom roles. Admin unhe banata hai kisi system role ko copy karke.
  • 03tenant_id = null — System roles ka tenant_id null hota hai. Matlab sab tenants ke liye available.

07 / 10
Registration Flow
Pehli baar register karne par kya kya hota hai — step by step.
  • KEYPura registration ek DB Transaction mein hota hai — ek bhi step fail ho to sab rollback ho jata hai. Partial state kabhi nahi banega.
1. Frontend — Form Submit
User form bharta hai: company_name, subdomain, email, password
POST /auth/register API call hoti hai
→ axios global instance use hota hai — manually URL nahi likhte
2. NestJS — ValidationPipe
Global ValidationPipe DTO validate karta hai automatically
Email format check, subdomain uniqueness check, password length check
→ ye globally registered hai — controller mein kuch nahi likhna
3. DB Transaction Start
TypeORM transaction start hoti hai — sab ek saath commit hoga
4. Tenant Row Create
tenants table mein naya record banta hai
config JSONB mein default stages, default locale, feature_flags (sab off) set hote hain
plan = 'free' by default
→ tenant ka unique ID generate hota hai (UUID)
5. User Row Create
users table mein pehla user banta hai
password bcrypt se hash hota hai — plain text kabhi store nahi hota
tenant_id = abhi banaye gaye tenant ka ID
6. System Roles Seed + Admin Assign
4 system roles create hote hain: admin, recruiter, viewer, client_spoc
user_roles table mein pehle user ko admin role assign hoti hai
JWT token generate hota hai — payload mein: { userId, tenantId, roles, permissions }
→ Transaction commit · Frontend ko token milta hai · Config fetch · Dashboard redirect

NestJS Backend Structure
Global middleware ek baar register hote hain. Feature modules apne andar sab kuch rakhte hain.
main.ts — Global Registrations
  • 01app.useGlobalPipes(ValidationPipe) — har DTO automatically validate hogi. Controller mein kuch nahi likhna.
  • 02app.useGlobalGuards(JwtAuthGuard, TenantGuard, RolesGuard) — har route protected. @Public() decorator se bypass kar sakte ho login jaise routes ke liye.
  • 03app.useGlobalInterceptors(TransformInterceptor) — har response ek fixed format mein aayegi: {"data": ..., "meta": ..., "status": 200}
  • 04app.useGlobalFilters(GlobalExceptionFilter) — koi bhi error kabhi raw nahi jayega. Standard error shape: {"error": ..., "message": ..., "code": ...}

Golden Rule
  • →Koi bhi feature module directly doosre module ka service inject nahi karega. Communication sirf EventEmitter2 events se hogi. Isse loose coupling rehti hai.
  • →Jab job banta hai → JobCreatedEvent emit hota hai → PipelineModule isko sun ke default pipeline bana deta hai. Jobs module ko pata nahi ki Pipeline exist bhi karta hai.

Next.js Frontend Structure
Ek global setup, har feature module isko use karta hai. Tenant config se dynamic UI.
lib/ — Global Setup (ek baar)
  • 01axios.ts — Ek axios instance banao. baseURL, timeout, token auto-attach, 401 pe logout — sab yahan. Poori app isi instance se calls karti hai.
  • 02query-client.ts — TanStack QueryClient globally configure hota hai. staleTime: 5min, retry: 2, global error handler. Har component isko import karta hai.
  • 03i18n.ts — Language setup ek baar. Tenant config se locale aata hai — en-IN, en-US wagera.

Complete Request Flow
Browser se Database tak — ek request ka poora safar.
BROWSER
User action — "Jobs list dekho"
acme.yourats.com/jobs pe navigate karta hai
NEXT.JS MIDDLEWARE (Edge)
Token check + tenant header
Cookie mein JWT nahi → redirect /login. JWT hai → x-tenant-slug: "acme" header add karo
TANSTACK QUERY + AXIOS
Cache check → API call
Cache fresh hai → cached data return. Stale hai → GET /jobs API call. Axios interceptor JWT header auto-attach karta hai.
NESTJS — GLOBAL GUARDS
JwtGuard → TenantGuard → RolesGuard
Token verify. tenantId resolve. jobs.read permission check. Ek bhi fail → 401/403 response, controller tak nahi pahunchta.
NESTJS — GLOBAL INTERCEPTOR
TransformInterceptor wraps response
Controller se data aata hai → interceptor use {"data": [...], "meta": {"total": 42}, "status": 200} mein wrap karta hai
JOBS CONTROLLER + SERVICE
@TenantId() se tenantId milti hai
Service ko tenantId milta hai → Repository WHERE tenant_id = ? AND deleted_at IS NULL lagata hai
POSTGRESQL
RLS + Query execute
Row Level Security extra safety layer. Sirf is tenant ke jobs return hote hain → Response back to frontend

Pipeline & Candidate Logic
Ek candidate multiple pipelines mein kaise hota hai — aur har jagah alag status.
  • 01Candidate tenant-level entity hai — matlab Rahul Kumar ek baar candidate table mein hoga. Uski profile ek baar store hogi.
